# chinese-poetry 项目报告

## 1. 项目定位与目标

`chinese-poetry` 是一个以 **JSON 为核心分发格式** 的中文古典诗文数据仓库，目标是为开发者、研究者和教育产品提供可直接复用的数据底座。仓库首页明确强调其覆盖规模（唐诗、宋诗、宋词及多个经典子集）与“便于二次开发”的定位。 

核心价值可概括为三点：

1. **规模化**：大体量诗词文本与作者信息，覆盖多个朝代与体裁。
2. **结构化**：统一采用 JSON，降低数据接入门槛。
3. **开源协作**：通过 PR/Issue 持续修订错字、异体字、元数据等问题。

## 2. 代码仓库结构总览

仓库在结构上可分为 4 层：

1. **原始/主数据层**：如 `全唐诗/`、`宋词/`、`五代诗词/`、`论语/`、`诗经/` 等。
2. **衍生特征层**：如 `rank/`（排序结果）、`strains/`（平仄模式）。
3. **加载与适配层**：`loader/data_loader.py` + `loader/datas.json`，提供面向下游任务的数据抽取机制。
4. **质量保障层**：`test_poetry.py` + GitHub Actions workflow（`pytest`）。

从维护角度看，该仓库是典型“**数据仓库 + 轻脚本**”模式，而非业务应用项目。

## 3. 数据资产盘点（当前仓库内容）

基于本地扫描统计，顶层中文目录共 14 个，主要数据集规模如下（JSON 文件数 / 记录数估算 / 体积）：

- 全唐诗：633 / 333,863 / 141.85 MB
- 御定全唐詩：900 / 43,103 / 19.72 MB
- 宋词：25 / 22,896 / 8.84 MB
- 元曲：1 / 11,057 / 3.99 MB
- 蒙学：13 / 13 / 1.01 MB（多为“单书单文件”结构）
- 其余子集（楚辞、论语、诗经、曹操诗集、纳兰性德等）为中小型专题库

### 3.1 数据形态特点

仓库并非单一 schema，而是“**统一 JSON 容器 + 多数据结构并存**”：

- 诗类主结构常见字段：`author/title/paragraphs/(id)`
- 词类常见字段：`author/rhythmic/paragraphs`
- 典籍类常见字段：`chapter/content/section`
- 衍生特征类常见字段：`strains`、`error_type` 等

这使它非常适合“语料拼接、检索、生成、统计分析”，但在“严格统一接口”场景需二次标准化。

## 4. 核心数据目录解读

### 4.1 `全唐诗/`

- 目录包含 `poet.tang.*.json` 和 `poet.song.*.json` 两大序列文件；README 标注每文件约 1000 条。
- 同时提供 `authors.tang.json` / `authors.song.json` 作者信息。
- README 明确指出：该目录存储为繁体（主要是全唐诗/全宋诗源数据）。
- 目录下还有 `error/` 子目录，保存包含 `error_type` 的纠错/平仄扩展版本。

### 4.2 `宋词/`

- `ci.song.*.json` 主体，README 说明为简体分发。
- 另有 `author.song.json`（词人信息）和 `ci.db`（SQLite 版本，README 标注“不保证持续维护”）。
- 字段以 `rhythmic`（词牌）+ `paragraphs` 为核心，适合词牌统计、风格建模。

### 4.3 `五代诗词/`

- 分为 `huajianji/` 与 `nantang/`。
- `huajianji` README 指出：可能与全唐诗/全宋词出现重复，这是历史时期与文献交叠导致的正常现象。

### 4.4 `蒙学/`

- 收录三字经、百家姓、千字文、弟子规、幼学琼林等启蒙读物。
- 该目录 README 提供了较充分的来源说明与结构示例，但不同书目之间 schema 差异大。

### 4.5 `strains/` 与 `rank/`

- `strains/README.md` 说明其数据与主诗词文件按文件名和数组索引一一对应，用于附加平仄等特征。
- 这种“侧表/外挂特征”设计保留了原始文本纯净性，也便于按需合并。

## 5. 数据加载机制分析

`loader/data_loader.py` 提供了 `PlainDataLoader`，由 `loader/datas.json` 驱动：

- 通过 `cp_path + datasets[*].path` 计算数据位置。
- 支持 **单文件** 与 **目录批量** 两种读取。
- 通过 `tag` 指定文本字段（如 `paragraphs/content/para`）。
- 支持按数据集名批量抽取、按 id 抽取。

### 5.1 优点

- **配置驱动**：新增数据集只需补 `datas.json`。
- **低耦合**：对下游调用者隐藏目录细节。

### 5.2 局限

- 默认假设 “目标 JSON 顶层为 list 且元素为 dict”。
- 对异常格式与巨大文件仅做最少处理（无流式读取/无容错策略）。
- `body_extractor` 内存拼接方式在超大规模任务下有压力。

## 6. 测试与质量保障

### 6.1 现有策略

- `test_poetry.py` 动态发现中文目录，对其中 JSON 执行递归加载校验（可解析即通过）。
- `.github/workflows/test.yml` 在 push/PR 时执行 `pytest`。

### 6.2 当前可观察问题

在当前环境中执行 `pytest -q` 失败，报错来自旧版 pytest 与 Python 3.10 的 AST 重写兼容问题（`TypeError: required field "lineno" missing from alias`），不是数据本身损坏。

直接执行 `python test_poetry.py` 可完成校验，说明 JSON 读取逻辑本身可运行。

## 7. 技术评估结论

### 7.1 项目优势

1. **数据规模与覆盖面领先**：适合训练语料、检索引擎、教育内容平台。
2. **开箱即用**：JSON 结构直观，Python/JS/Go 等可直接消费。
3. **分层清晰**：主数据与衍生特征（rank/strains）分离。
4. **可配置加载**：`datas.json` + `PlainDataLoader` 降低接入成本。

### 7.2 主要风险

1. **Schema 不统一**：跨目录聚合时需做字段映射。
2. **数据重复风险**：跨子集存在文献重叠（README 已提醒）。
3. **测试栈老化**：`pytest==5.3.2` 对较新 Python 版本兼容不足。
4. **元数据治理不足**：缺统一版本号、来源追踪、字段字典。

## 8. 改进建议（按优先级）

### P0（优先立刻做）

1. **升级测试依赖**：将 pytest 升级至与当前 Python 版本兼容的主流版本。
2. **补充统一数据字典文档**：建立字段映射表（author/title/paragraphs/content/rhythmic/...）。
3. **为 loader 增加异常与类型校验**：对非 list 顶层、缺 tag、坏文件给出清晰报错。

### P1（短期）

1. **增加数据统计脚本**：自动输出每个目录文件数、记录数、坏文件数、schema 分布。
2. **增加“去重工具链”**：按标题+作者+文本 hash 标记疑似重复，供人工复核。
3. **引入最小 API 层**：提供统一查询接口（按作者/朝代/词牌/关键字）。

### P2（中期）

1. **引入版本化数据发布**（例如 release + changelog）。
2. **建立来源可追溯机制**（source_url/source_note/revision）。
3. **补充高质量基准子集**（严格校勘、教育场景可直接引用）。

## 9. 适用场景建议

该仓库适合以下场景：

- **NLP/LLM 训练与评测**：古汉语生成、风格迁移、文本检索。
- **教育产品**：诗词学习、诵读、释义配套。
- **文化可视化**：词牌热度、作者关系、朝代主题演化。
- **工具型应用**：每日诗词、桌面组件、诗词日历。

对“需要强一致 schema 与事务级数据质量”的生产系统，建议在该仓库之上构建一层标准化 ETL。

## 10. 总结

`chinese-poetry` 的核心竞争力不是复杂代码，而是**稀缺且可计算的中文古典文本资产**。当前技术栈简洁，适合作为“上游数据源”；若进一步完成 schema 治理、测试现代化、可追溯发布，它可以从“优秀开源语料仓库”进化为“稳定的数据基础设施”。
